# 프로젝트 구조 가이드 — Visual Studio (Client / Server / Shared)

> **한 솔루션(`SpaceWar.slnx`) 안에 프로젝트 3개.** 각 프로젝트에 무엇을 넣고, Shared를 어떻게 쓰는지 정리한 문서.
> 대상: 이태형(클라·렌더) / 안병규(클라·게임플레이) / 한현우(서버) / 지도교수님

---

## 1. 전체 구조

```
Project/SpaceWar/
├── SpaceWar.slnx          솔루션 (프로젝트 3개를 묶음)
├── Client/                🎮 클라이언트  (Client.exe)
│   ├── Client.vcxproj
│   └── main.cpp
├── Server/                🖧 서버        (Server.exe)
│   ├── Server.vcxproj
│   └── main.cpp
└── Shared/                🔗 공용 라이브러리 (Shared.lib)
    ├── Shared.vcxproj
    ├── Protocol.h
    └── Shared.cpp
```

- **Client / Server** = 각각 실행파일(.exe)
- **Shared** = 정적 라이브러리(.lib) — 클라·서버가 **똑같이** 링크
- 빌드 설정: **x64 전용** · 툴셋 **v145(VS 2026)** · **C++ 최신(stdcpplatest)**
  - x86(Win32)은 뺐습니다. DX12/DXR·대규모 서버는 x64만 쓰면 됩니다. (필요하면 나중에 추가 가능)

### 의존 방향 (중요)
```
   Client ──┐
            ├──►  Shared      ✅ 클라·서버는 Shared를 참조
   Server ──┘

   Client  ✗  Server          ❌ 클라와 서버는 서로 직접 참조하지 않는다
```

---

## 2. 프로젝트별 — 무엇을 넣나

### 🔗 Shared (정적 라이브러리) — 가장 먼저 이해할 것

**"클라와 서버가 반드시 똑같이 알아야 하는 것"의 단일 원본.**

| ✅ 넣을 것 | ❌ 넣지 말 것 |
|-----------|--------------|
| 네트워크 **패킷 구조체 / 프로토콜** (`Protocol.h`) | **DirectX** 코드 (`d3d12.h`, `ComPtr` …) |
| 공용 **데이터 구조 / enum / 상수** | **IOCP / 소켓 구현** (그건 Server) |
| 공용 **수학 타입**(벡터·행렬 등) | **렌더링 / 게임플레이 전용** 로직 |
| (선택) 서버 권위 검증에 쓰는 **공용 시뮬레이션 규칙** | 특정 한쪽만 쓰는 것 전부 |

> 💡 **왜 중요한가:** 패킷 포맷을 여기 한 곳에만 두면, 고칠 때 **클라·서버가 동시에 반영**됩니다. 서로 다른 패킷 포맷을 쓰다 생기는 네트워크 버그(1순위)를 원천 차단.

### 🖧 Server (콘솔 exe) — 한현우

- **넣는 것:** IOCP 서버(리슨·워커 스레드·AOI 브로드캐스트), 권위 판정(위치·중력·발사·무력화 검증), 매치/소대 관리
- **안 넣는 것:** DirectX. **서버는 그래픽을 모른다.** (GPU 없이 돌아야 함)
- 링크: `ws2_32.lib`(Winsock) 등. **DirectX·FMOD·Assimp 링크 금지.**

### 🎮 Client (윈도우 exe) — 이태형 · 안병규

- **넣는 것:** DX12/DXR 렌더러, 게임플레이, 입력, 사운드(FMOD), 리소스 로딩(Assimp)
- **DirectX·FMOD·Assimp·DLSS·NRD 링크는 여기(Client 프로젝트)에만** 추가한다.
- **내부는 렌더러 경계 명세대로 폴더/클래스로 나눈다** (같은 exe 안에서 코드로 분리):

```
Client/
├── Engine/         재사용 코어
│   ├── Graphics/   ★ DirectX는 여기 안에만 (GRenderer, DXR, Denoise, Upscale)
│   ├── Resource/   Assimp 래퍼
│   ├── Audio/      FMOD 래퍼
│   ├── Scene/      노드 SoA (DX 모름)
│   └── ...
├── Game/           게임플레이 (핸들·Scene만, DX 모름)
└── main.cpp
```
> 렌더/게임플레이 담당이 다르니(이태형/안병규), `Engine/Graphics`(이태형)와 `Game/`(안병규)로 나누면 같은 파일 충돌도 줄어든다.

## 3. 코드에서 Shared 쓰는 법

Client·Server 프로젝트는 **포함 경로에 솔루션 폴더가 잡혀 있어서**(`AdditionalIncludeDirectories = $(ProjectDir)..\`) 이렇게 쓰면 됩니다:

```cpp
#include "Shared/Protocol.h"

void OnRecv() {
    Shared::PlayerMovePacket pkt{};
    pkt.header.type = Shared::PacketType::PlayerMove;
    // ... 클라와 서버가 똑같은 구조체로 주고받음
}
```

- **Shared에 새 헤더를 추가**하면 클라·서버 양쪽에서 즉시 `#include` 가능.
- Shared는 이미 **프로젝트 참조(Project Reference)** 로 연결돼 있어, 링크는 자동으로 됩니다.

---

## 4. 소스 파일 추가하는 법 (VS)

1. **솔루션 탐색기**에서 해당 프로젝트(Client/Server/Shared) 우클릭 → **추가 → 새 항목**
2. `.h`(헤더) / `.cpp`(소스) 선택, 이름 지정
3. 내부 폴더를 만들려면 프로젝트 우클릭 → **추가 → 새 필터**(가상 폴더) 또는 실제 폴더를 만들고 파일을 그 안에 생성
   - 실제 폴더 구조를 쓰면 위 Client 내부 구성처럼 물리적으로도 정리됩니다.

> ⚠️ **DirectX/FMOD/Assimp 라이브러리 연결은 Client 프로젝트 속성에서만** 하세요.
> (프로젝트 속성 → VC++ 디렉터리 / 링커 → 입력. SDK는 `Project/external/`에 두고 경로 지정)

## 5. 빌드 & 실행

- **구성:** 상단 툴바에서 **x64** 선택 (Debug/Release)
- **시작 프로젝트 지정:** 솔루션 우클릭 → 시작 프로젝트 설정
  - 단일: Client 또는 Server를 시작 프로젝트로
  - **다중(로컬 테스트용):** "여러 개의 시작 프로젝트" → **Server + Client 둘 다 시작** → F5 한 번에 서버 켜고 클라 붙이기. 50v50 테스트 때 클라 여러 개 실행도 가능.

---

## 6. 규칙 체크리스트

- [ ] `#include <d3d12.h>` 는 **Client 프로젝트 안에서만** (Server·Shared 금지)
- [ ] 패킷·공용 구조체는 **Shared 한 곳에만** 정의
- [ ] Client 와 Server 는 **서로 직접 참조하지 않는다** (연결은 오직 Shared를 통해)
- [ ] DirectX·FMOD·Assimp 링크는 **Client 프로젝트에만**
- [ ] 구성은 **x64**로 통일
- [ ] 빌드 산출물 폴더(`x64/`, `.vs/` 등)는 커밋하지 않는다 (`.gitignore` 처리됨)

## 7. 참고 — 나중에 더 쪼갤 수 있음

클라가 커지면 **렌더러를 `ClientEngine.lib`(정적 라이브러리)** 로 따로 빼서, 게임플레이(안병규) 코드가 DX 헤더를 아예 링크 못 하게 **빌드 레벨로** 경계를 더 강하게 만들 수도 있습니다. 지금은 **Client 한 프로젝트 안에서 폴더·클래스(GRenderer/Scene)로 나누는 것**으로 충분합니다.

---

> **한 줄 요약:** Client·Server 는 각자 exe, Shared 는 둘이 공유하는 lib. **DX는 Client에만, 프로토콜은 Shared에만, 클라↔서버 직접 참조는 금지.**
