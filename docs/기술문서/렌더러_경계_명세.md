# 렌더러 경계 설계 명세 — DirectX를 렌더러 안에 가두기

> **문서 목적:** 프레임워크에서 가장 먼저 정해야 할 결정 —
> "DirectX 관련 코드를 렌더러에만 두고, 게임 로직은 그걸 전혀 모르게" 하는 **경계**를 정의한다.
> **대상 독자:** 이태형(렌더링·주 작성) / 지도교수님 / 안병규(게임플레이)
> **상태:** 초안 · 2026-07-13 · 방식 **A(Scene 계층형)** 채택

---

## 0. 이 문서가 정하는 한 가지

> **게임 로직(게임플레이·서버)은 DirectX를 절대 보지 않는다. DX는 렌더러 안에만 존재한다.**

이걸 **가장 먼저** 정하는 이유: 한 번 DX 코드가 게임 로직에 새어 들어가면 나중에 떼어내는 건 사실상 전면 재작성이다. 반대로 이 경계만 잡아두면, 앞으로 붙일 **DXR·Path Tracing·NRD·DLSS·적응형 샘플링**은 전부 **렌더러 내부 교체**로 끝나고 게임플레이·서버는 아무것도 안 바뀐다. (우리 렌더러는 연구 주제라 계속 갈아엎힐 것이므로 이 격리가 특히 중요하다.)

---

## 1. 핵심 원칙 (한 문장)

**게임 로직은 "핸들(번호표) + 고수준 명령"만 다루고, `ID3D12*`·`ComPtr` 같은 DX 타입은 절대 만지지 않는다.**

- ✅ 게임플레이가 하는 말: *"이 메쉬를 이 위치·자세로 그려줘"*
- ❌ 게임플레이가 절대 안 하는 것: 커맨드리스트 기록, 리소스 배리어, PSO 바인딩, 힙 관리 …

---

## 2. 3계층 구조

```
┌─────────────────────────────────────────────┐
│  게임플레이 로직 (안병규) / 서버(한현우)          │  DX 전혀 모름
│   - 엔티티·규칙·입력 처리                        │  핸들과 트랜스폼만 다룸
└───────────────┬─────────────────────────────┘
                │  AddNode / SetTransform (핸들 기반 API)
                ▼
┌─────────────────────────────────────────────┐
│  Scene (중간 계층)                             │  DX 없음 (DX-free)
│   - 노드 계층 소유, 월드 변환 계산              │
│   - 컬링 → RenderItem 목록 추출·정렬            │
└───────────────┬─────────────────────────────┘
                │  RenderItem[] (POD 목록, DX 타입 없음)
                ▼
┌─────────────────────────────────────────────┐
│  GRenderer (이태형)                            │  ★ DX는 여기 안에만 ★
│   - device / swapchain / queue / command      │
│   - PSO · DXR · NRD · DLSS · Path Tracing      │
└─────────────────────────────────────────────┘
```

### 각 계층이 아는 것 / 모르는 것

| 계층 | 아는 것 | 모르는 것 |
|------|---------|-----------|
| **게임플레이 / 서버** | 핸들(숫자), 트랜스폼(행렬), 게임 규칙 | DirectX 존재 자체 |
| **Scene** | 노드 계층, 변환, 컬링, 렌더 목록 | DX API (커맨드·PSO·힙 등) |
| **GRenderer** | DX 전부, 그리는 방법 | 게임 규칙, 씬이 왜 그렇게 생겼는지 |

> 핵심: **Scene은 DX를 모르는 "중간 통역사"** 다. 게임플레이가 준 정보를 렌더러가 먹기 좋은 평평한 목록으로 바꿔줄 뿐, 자기도 DX는 모른다.

---

## 3. 경계를 넘는 것 vs 절대 안 넘는 것

| 경계를 **넘어도 되는 것** | 경계를 **절대 넘으면 안 되는 것** |
|--------------------------|----------------------------------|
| 핸들 (`MeshHandle`, `NodeHandle` 등 = 그냥 정수) | `ID3D12Device`, `ID3D12Resource` … |
| 트랜스폼/수학 타입 (`Matrix`, `Vector3`) | `ComPtr<...>` (COM 스마트포인터) |
| `RenderItem` 같은 POD 구조체 | 커맨드리스트 / 커맨드큐 |
| 파일 경로, 설정 값 | 리소스 배리어, 디스크립터 힙, PSO 객체 |

**판단 기준 한 줄:** *게임플레이 헤더(`.h`)에서 `#include <d3d12.h>`가 필요해지면, 그 순간 경계가 깨진 것이다.*

## 4. 인터페이스 스케치

> 실제 시그니처가 아니라 **모양**을 보여주는 스케치다. 이름은 기존 코드(GRenderer 등)에 맞춰 조정 가능.

```cpp
// ── 경계를 넘는 것: 불투명 핸들 (그냥 "번호표") ─────────────
using MeshHandle     = uint32_t;
using MaterialHandle = uint16_t;
using NodeHandle     = uint32_t;

// ── 게임플레이(안병규)가 보는 전부: DX 타입이 하나도 없다 ──────
class Scene {
public:
    NodeHandle AddNode(NodeHandle parent, MeshHandle mesh, MaterialHandle mat);
    void       SetLocalTransform(NodeHandle n, const Matrix& local); // 매 프레임 이것만
    void       SetMesh(NodeHandle n, MeshHandle mesh);
    void       Remove(NodeHandle n);
    // 카메라, 가시성 토글 등도 여기에
};

// ── 리소스 로딩도 DX를 숨긴다 ──────────────────────────────
class ResourceManager {
public:
    MeshHandle     LoadMesh(const std::string& path);   // 내부: Assimp 파싱 + GPU 업로드(숨김)
    MaterialHandle LoadMaterial(/* ... */);
};

// ── Scene이 렌더러에 넘기는 것: POD, DX 타입 없음 ──────────
struct RenderItem {
    uint64_t       sortKey;   // 정렬용(패스·머티리얼·메쉬·깊이 패킹)
    NodeHandle     node;      // 월드행렬 참조용
    MeshHandle     mesh;
    MaterialHandle material;
};

// ── 렌더러: DirectX는 전부 이 클래스 "안"에만 ─────────────
class GRenderer {
public:
    void BeginFrame();
    void Render(const RenderView& view);   // Scene이 추출한 RenderItem[] 소비
    void EndFrame();                        // Present
private:
    // ↓↓↓ 아래는 게임플레이·서버가 절대 못 봄 ↓↓↓
    // ComPtr<ID3D12Device> device;  CommandQueue;  SwapChain;
    // PSOTable;  DXR;  NRD;  DLSS;  PathTracer; ...
};
```

### 한 프레임 흐름

```
게임플레이:  scene.SetLocalTransform(...)      // 위치만 갱신
Scene:       월드행렬 계산 → 컬링 → RenderItem[] 추출·정렬
GRenderer:   RenderItem[]를 받아 DX로 그림 → Present
```

게임플레이는 **① 핸들 받기 ② 트랜스폼 갱신** 이 사실상 전부다. 나머지는 Scene과 GRenderer가 알아서 한다.

## 5. 풀어서 — 헷갈리기 쉬운 개념 3가지

### ① "핸들"이 뭐고 왜 포인터 대신 쓰나
핸들은 **번호표(정수)** 다. 실제 GPU 메쉬 데이터는 렌더러가 금고에 넣어두고, 게임플레이에는 **"37번 메쉬"** 같은 번호만 준다. 게임플레이는 그 번호로 "37번 그려줘"라고만 말한다.
- **왜 좋은가:** 게임플레이가 실제 DX 객체(포인터)를 들고 있으면 DX를 알아야 하고, 렌더러가 내부를 바꾸면 다 깨진다. 번호표만 들고 있으면 렌더러가 안에서 뭘 하든 상관없다. → **경계가 지켜진다.**

### ② 왜 "계층"과 "렌더 목록"을 나누나
- **계층(부모-자식)** 은 *"우주선에 붙은 포탑", "플레이어가 든 무기", "어깨 너머 카메라"* 처럼 **누가 누구를 따라 움직이는지** 계산할 때만 쓴다.
- 하지만 **실제로 그릴 때**는 트리를 타고 다니지 않고, **평평한 목록**만 쭉 훑는다.
- 쉽게: *족보(계층)로 위치를 계산해두고, 그림 그릴 땐 명단(목록)만 본다.* 트리를 매번 뒤지는 것보다 훨씬 빠르고, 100인 난전처럼 오브젝트가 많을수록 이 차이가 커진다.

### ③ Path Tracing·디노이저는 왜 "안정적인 번호표"를 원하나
PT·NRD·DLSS는 노이즈를 지우려고 **"이번 프레임 이 픽셀이 지난 프레임엔 어디 있었나"** 를 추적한다(= 시간적 누적).
- 이게 되려면 오브젝트가 **프레임이 지나도 같은 번호표(핸들)** 를 유지해야 한다. "37번은 어제도 오늘도 그 우주선"이어야 이전 위치와 비교해 모션벡터를 만든다.
- Scene 계층형은 핸들이 영속적이라 이게 **공짜로** 된다. (매 프레임 목록을 새로 만드는 방식이었으면 번호가 매번 바뀌어 직접 관리해야 했다.)

## 6. 왜 이 구조(A)인가 (요약)

| 우리 프로젝트 특성 | A가 맞는 이유 |
|--------------------|----------------|
| 렌더러가 계속 진화 (PT 연구) | 렌더러 = "단순 소비자" → 내부를 갈아엎어도 나머지 무손상 |
| PT·NRD·DLSS 등 시간적 기술 | 영속 핸들 = 안정 ID·모션벡터 공짜 |
| 탈것·무기·숄더캠 | 부모-자식 계층 내장 |
| 클라 분업 (렌더/게임플레이) | 안병규는 핸들만 → DX 완전 무지, client1/client2 경계와 일치 |

> 대안 B(렌더러가 씬까지 관리)는 *자주 바뀔 것*과 *안정적이어야 할 것*이 엉켜서 탈락, C(매프레임 제출)는 시간적 안정 ID가 사라져 PT와 안 맞아 탈락.

## 7. 단계적 구현 (얇게 시작)

목표 구조는 A로 잡되, 처음부터 다 만들지 않는다.

- **Phase 1 (토대):** 노드 배열 + 월드 변환 계산 + **순회하며 즉시 그리기.** 삼각형·메쉬가 화면에 뜨는 것까지.
- **Phase 2 (수집):** 컬링 → RenderItem 추출 → 정렬. 렌더 목록 분리.
- **Phase 3 (성능):** 인스턴싱·배칭, 멀티스레드 커맨드 기록.

> 중요한 건 **핸들 기반 경계와 3계층 골격을 Phase 1부터 지키는 것.** 그러면 Phase 2·3은 게임플레이 코드를 안 건드리고 얹을 수 있다.

## 8. 남은 결정 (열린 항목)

- **A1 vs A2 — 게임플레이 엔티티와 씬 노드의 관계**
  - **A1:** 게임플레이 엔티티가 곧 씬 노드다 (단순, 결합 강함)
  - **A2:** 게임플레이 엔티티는 따로 있고, 매 프레임 씬으로 트랜스폼만 추출·동기화 (분리 깔끔, 서버 권위·시뮬레이션 분리와 잘 맞음)
  - *잠정 방향:* 서버 권위 구조상 **A2**가 유력하나, 확정 전 논의 필요.
- **네이밍 확정** — `GRenderer` / `Scene` / `ResourceManager` 등 기존 코드와 정합성 맞추기.
- **머티리얼 표현 범위** — 하이브리드 PT에서 머티리얼을 어디까지 핸들로 추상화할지.

---

## 9. 지켜야 할 규칙 (체크리스트)

- [ ] 게임플레이·서버 코드에서 `#include <d3d12.h>` / `ComPtr` **금지**
- [ ] 게임플레이는 **핸들 + 트랜스폼**만으로 대화한다
- [ ] DX 객체(디바이스·커맨드·PSO·힙)는 **GRenderer 밖으로 안 나간다**
- [ ] Scene은 DX를 모른다 (계산·목록 구성만)
- [ ] 리소스 로딩(Assimp·업로드)은 `ResourceManager`가 DX를 숨긴 채 핸들만 반환
- [ ] 새 렌더 기능(RT·디노이즈·업스케일)은 **렌더러 내부에서만** 추가

---

> **한 줄 요약:** 게임 로직은 번호표(핸들)로만 말하고, DirectX는 렌더러 금고 안에서 절대 안 나온다.
